# Rules Synthesis

This closes the learning loop. A single coaching note helps one developer on one PR; this skill turns a pile of them into shared rules that change how the AI works with the *whole team*, so recurring friction gets designed out instead of re-coached every time.

## Input
Read `docs/agents/coaching-notes/`. Each note carries a `signals` block with tagged dimensions and `rating` values (`strong` / `solid` / `growth_area`). Parse those for counts, and read the prose for the *why* behind a pattern.

## Analysis
- **Aggregate by dimension.** Count `growth_area` ratings per dimension across all notes. A dimension that recurs across many PRs and many people is a team-level pattern worth a rule. A one-off is not — wait for a pattern.
- **Look per developer too**, but as growth, not ranking — to spot where someone could use support, or where a strength is worth sharing. Keep it constructive and private by default.
- **Read for nuance.** Two notes tagged `spec-clarity / growth_area` may have different root causes (missing acceptance criteria vs. missing error cases). A good rule targets the real cause.

## Output
**1. A trends summary** for the human: what's improving, what recurs, the top few patterns with evidence — constructive in tone.

**2. The versioned `docs/agents/team-rules.md`**, written *to a future AI*: how to collaborate better with these developers given what's been seen. Rules can be behavioral *and* skill-shaped — including telling the agent which skill to invoke. Each rule pairs guidance with a one-line rationale.
```
# Team Collaboration Rules
<!-- Generated by rules-synthesis on <date> from <N> coaching notes. Version <n>. -->

## Before building
- Run grill-with-docs and confirm acceptance criteria and error cases before any code.
  *Why: spec-clarity is the top growth area — ~60% of recent PRs reworked from under-specified requirements.*

## During implementation
- Surface architecture and tradeoff decisions to the developer instead of deciding silently.
  *Why: decision-ownership recurs as a growth area; developers want these calls.*
```

## How the loop closes
`grill-with-docs` and `to-prd` read this file to seed their alignment checks, and the orchestrator reads it at step 0. So updating it changes what the *next* feature's alignment focuses on — yesterday's coaching becomes today's sharper grilling. Point this out to the user; the rules only matter if the skills are aimed at the same file.

## Cadence & caution
- Run weekly, or after ~10–20 PRs — patterns need volume.
- Don't over-fit to a handful of notes; with only a few, say the sample is small and keep rules tentative.
- Re-synthesize as notes accumulate; bump the version and note what changed.
