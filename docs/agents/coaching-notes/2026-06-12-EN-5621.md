# Coaching note — EN-5621 / PR #25
*2026-06-12 · ADR 0001 / S3 — Shared orchestration-spine spec + facilitation.md in core*

## What shipped

Two new reference files in `skills/core/core-setup/`:
- `orchestration-spine.md` — shared conductor contract (5 elements)
- `facilitation.md` — AFCI + PDF-Loop reference patterns

`README.md` updated: `core-setup` marked ★ with links to all three reference files.

## Efficiency

Clean single-pass implementation. The main deliberation cost was the file-placement question — whether `orchestration-spine.md` should live at the `skills/core/` group root or inside `core-setup/`. The decision to put it in `core-setup/` (alongside the existing `MCP-SETUP.md`) was the right call: consistent with the established reference-doc pattern, gets installed when the `core` group installs, and lets conductors reference it by the same `core-setup/` path convention already in use.

The `config.md` key `verifier.max_fix_attempts` for the bounded auto-fix loop N was introduced in the spec — this is a new key that `core-setup`'s config shape will need to document and that future conductor implementations should read. Worth a check-in before S4 wires conductors to the spine.

## What this story didn't do (by design)

The ADR migration list includes "Update `fe-flow` align step + PM conductors to reference `core-grill` and the spine spec" — that's deferred to S4+. The spine spec exists now as the single source of truth; the wiring comes next.

## Collaboration pattern

Interactive mode with a single implementation decision check-in (file placement) before writing, and one question after implementation (`verifier.max_fix_attempts` key name). Both were genuine forks worth pausing on. The scope stayed tight — no PRD, no slicing, no alignment stage needed for a pure doc story.

## For core-distill-rules

- Reference docs placed in a skill's directory (alongside `SKILL.md`) follow the `core-setup/MCP-SETUP.md` pattern — this is the established convention for shared cross-skill reference material in the `core` group.
- New `config.md` keys introduced in a spec doc should be noted in the commit message or PR description so the next story knows to wire them in.
