# Coaching note — EN-5602: wire publish skills to shared MCP-SETUP.md protocol

**What went well**

- **Spec was airtight.** The issue carried clear, checkable acceptance criteria and explicit context about what had already merged (EN-5600, EN-5601). There was no gap-check required — the description answered every question before it was asked. That's the gold standard for handoff to an agent.
- **Scope discipline was perfect.** Three files, additive changes only, no scope creep. Handing the agent a small, well-bounded task made the whole cycle fast and predictable.
- **Context pointer was exactly right.** Saying "reference MCP-SETUP.md sections rather than duplicating" meant the agent didn't have to make a judgment call about whether to inline vs reference — the decision was already made. That's decision-ownership working well.

**Opportunities for next time**

1. **PR body via shell with special characters.** Passing a multi-line string containing `§`, backticks, and markdown through the shell's `-c` flag or a heredoc caused quoting failures and several wasted turns. This isn't a spec or collaboration gap — it's a known shell hazard. The fix (Python-written temp file + `--body-file`) worked cleanly. Next time, use `--body-file` from the start whenever the PR body will contain special characters or markdown formatting.

**Bottom line**

This was a well-run cycle — tight spec, right context, right scope. The only friction was a mechanical tooling stumble on PR creation, not anything about how the work was framed or driven. The loop is working: EN-5599 is close to done, the substrate is cleaner, and the pattern of "define once, reference everywhere" is holding.
